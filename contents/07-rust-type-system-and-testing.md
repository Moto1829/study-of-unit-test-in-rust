# Rustの型システムと単体テスト

Rustの型システムは、単にコンパイル時の安全性を高めるだけではなく、テストの書きやすさと信頼性にも直結します。

この章の重要点は、型システムがテストを不要にするという話ではないことです。むしろ、型が守るべきことを静的に押さえてくれるため、単体テストはより本質的な仕様確認に集中できるようになります。

## コンパイル時に不正な状態を減らせる

型で表現できる制約が強いほど、実行時に確認すべきケースが減ります。たとえば、`Option` や `Result` が明示されていれば、失敗可能性や値の欠如を見落としにくくなります。その結果、何をテストすべきかが明確になります。

たとえば、nullを自由に取り得る設計だと、至るところでnull関連の防御とテストが必要になります。一方、Rustでは「値がない可能性」を `Option` として明示するため、考慮漏れがコード上に現れやすくなります。これはテストケース設計の指針としても非常に強力です。

## trait による差し替えがしやすい

依存先をtraitで抽象化しておけば、本番実装とテスト実装を明確に分離できます。これはスタブやモックの導入を自然にし、依存が多い処理でも小さな単位で検証しやすくします。

traitは単なるインターフェースではなく、「このユースケースが外部に何を要求しているか」を最小限の形で表す契約です。契約が小さく明確であるほど、テストダブルも簡潔になります。

## enum により分岐の網羅性を意識しやすい

状態や結果をenumで表現すると、分岐漏れに気付きやすくなります。どのバリアントをテストすべきかが整理しやすく、異常系の抜け漏れ防止にもつながります。

たとえば、処理結果を `Success`、`ValidationError`、`Conflict` のようなenumで表現していれば、テストケースもそのバリアントに沿って整理しやすくなります。文字列や数値で状態を表すより、仕様の見通しが大きく改善します。

## 所有権と借用が副作用の境界を明確にする

Rustではデータの所有者と参照の関係が明確なため、どこで状態が変化し得るかを追いやすくなります。これはテスト対象の責務を絞り込みやすくし、テストケースの設計を安定させます。

副作用が広がるコードは、観測対象が増え、テストも不安定になりがちです。所有権と借用の制約はその広がりを抑えやすく、結果として「何を見れば十分か」を決めやすくします。

## 設計の段階でテストしやすい形に寄せやすい

Rustでは型とインターフェースを丁寧に設計することが、そのままテストしやすい構造につながりやすい傾向があります。結果として、後から無理にテストを当てるのではなく、最初からテスト可能な設計を選びやすくなります。

たとえば、入力と出力が明確な純粋関数に寄せる、外部依存をtraitで隠す、状態遷移をenumで表す、といった設計は、そのままテストケースの明瞭さにつながります。

## 型が守ることとテストが守ること

切り分けの目安としては次の通りです。

- 型が守ること: 値の存在有無、状態の種類、呼び出し契約の不一致
- テストが守ること: ビジネスルール、境界条件、組み合わせ条件、外部依存との連携仕様

この切り分けが曖昧だと、型で防げることまでテストで重複確認したり、逆に仕様確認を型だけに期待して抜け漏れが出たりします。

## 型システムを活かす視点

- 不正な状態を型で表現不能にする
- 成功と失敗を `Result` で明示する
- 状態のバリエーションをenumに閉じ込める
- 依存をtraitで抽象化する
- 関数の責務を小さくし、入出力を明確にする

さらに、次のような設計も有効です。

- 文字列ではなく専用型で意味を表す
- 不正な組み合わせをコンストラクタで禁止する
- 可変状態を必要最小限に閉じ込める
- 失敗理由を雑なboolではなく型で表現する

## よくある誤解

- 型が強いからテストはいらない
- コンパイルが通るなら仕様も正しい
- traitを使えば自動的にテストしやすい
- enum化すれば異常系を全部網羅できる

実際には、型は非常に強力ですが、仕様の正しさそのものはテストで確認する必要があります。重要なのは、型とテストの責務を重ねるのではなく、補完関係として設計することです。

## この章の要点

- 型システムはテストを減らすというより、焦点を明確にする
- `Option`、`Result`、enumはテストケース設計を整理しやすくする
- traitは差し替え可能性と責務分離の両方に効く
- 所有権と借用は副作用境界を把握しやすくする
- 型で守ることとテストで守ることを分担させるのが重要

こうした設計は、テストケースの削減ではなく、必要なテストを明確にする方向に効きます。これがRustにおける型とテストの大きな相乗効果です。
