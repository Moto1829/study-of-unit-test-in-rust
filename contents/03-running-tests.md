# 単体テストの実行方法

テストコードは書くだけでは不十分で、効率良く実行し、失敗原因をすばやく把握できることが重要です。

とくにRustでは、コンパイルエラーとテスト失敗の両方が短いサイクルで返ってくるため、実行の仕方を理解しているかどうかで開発速度がかなり変わります。

## 基本

Rustでは通常、`cargo test` を使ってテストを実行します。

ここで押さえるべき点は次の通りです。

- すべてのテストを実行する方法
- 特定のテスト名だけを対象にする方法
- 標準出力を表示しながらテストする方法
- 失敗時のログをどう読むか
- 開発中に高速に回すための運用

代表的な実行例は次の通りです。

```bash
cargo test
cargo test add_returns_sum
cargo test -- --nocapture
```

1つ目は全テスト実行、2つ目は名前に一致するテストだけ実行、3つ目は標準出力を表示しながらの実行です。

## `cargo test` の実務上の使い分け

### 全体確認

変更の区切りでは全テストを実行し、他の箇所へ影響していないかを確認します。

### 部分実行

実装中は対象モジュールや対象テストに絞って回すと効率が上がります。修正中の箇所だけを高頻度で回し、最後に全体を確認する流れが現実的です。

### 出力確認

デバッグ目的で `println!` を使うときは、`-- --nocapture` を併用しないと出力が見えません。ただし、恒常的に出力へ頼るより、アサーションを明確にしたほうが保守しやすくなります。

## 実行時に意識したいこと

### 小さく回す

大きな変更をしてから一気に実行するより、変更単位ごとに小さく回したほうが原因特定は容易です。

この運用は、単に速いからではなく、失敗原因の候補が少ない状態を維持できるから有効です。直前の変更だけを疑える状態を保つのが基本です。

### 失敗メッセージを仕様として読む

テストの失敗は単なるエラーではなく、仕様と実装のずれを示す情報です。アサーションのメッセージや比較対象を明確にしておくと、修正コストが下がります。

たとえば、値が違ったのか、エラーになるべき場面で成功してしまったのか、あるいは逆に成功すべき場面で失敗したのかによって、見るべき原因は変わります。失敗メッセージを読む習慣は、デバッグ効率に直結します。

### 並列実行の影響を理解する

Rustのテストは並列で動くことがあります。共有状態や外部I/Oを使うテストでは、順序依存が入り込まないよう注意が必要です。

とくに次のようなケースは不安定になりやすいです。

- 同じファイルパスを複数テストが使う
- グローバル状態を書き換える
- 環境変数を変更する
- 時刻や乱数に依存する

こうしたテストは、状態を局所化するか、依存を抽象化して制御可能にする必要があります。

## 失敗時の見方

テスト失敗時は、まず以下を切り分けます。

1. コンパイルに失敗しているのか
2. テスト本体がpanicしているのか
3. アサーションが失敗しているのか
4. 前提条件の準備に失敗しているのか

この切り分けができると、コードの不具合なのか、テストの設計が悪いのか、環境依存なのかを素早く判断できます。

## 実行を安定させるための工夫

- テスト間で状態を共有しない
- テストデータを都度作る
- 外部I/Oはできるだけ隠蔽する
- 失敗時に読み取れるアサーションを書く
- 不安定な値に依存しない

安定しないテストは、価値が低いだけでなく、開発者がテストを信用しなくなる原因になります。速さだけでなく再現性も重要です。

## 開発フローとの関係

単体テストは、実装後にまとめて書く運用もできますが、小さな仕様単位で実装とテストを往復するほうが安定しやすいです。

とくにRustではコンパイラが強く設計を支えてくれるため、コンパイルエラーとテスト失敗を組み合わせることで、かなり早い段階で問題を絞り込めます。

## この章の要点

- `cargo test` は全体確認と部分実行を使い分ける
- 失敗ログは仕様との差分として読む
- 並列実行による共有状態の問題を軽視しない
- 小さく回す運用が最も原因特定しやすい
- 再現性の低いテストは早めに設計から見直す
