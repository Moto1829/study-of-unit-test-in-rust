# 単体テストの作り方

Rustの単体テストは、対象コードと同じモジュール内、または近い単位で記述することが多く、内部実装に近い粒度で振る舞いを検証しやすいのが特徴です。

統合テストが公開インターフェース全体の整合性を見るのに対し、単体テストは小さな責務の正しさを素早く確認するためのものです。実装のごく近くに置けるため、変更に追従しやすく、失敗時の原因も追いやすい利点があります。

## 基本構文

単体テストでは主に以下を使います。

- `#[cfg(test)]`
- `mod tests`
- `#[test]`
- `assert!`
- `assert_eq!`
- `assert_ne!`

`#[cfg(test)]` はテスト時だけコンパイルしたいコードを囲うために使います。`#[test]` が付いた関数はテストランナーから実行されます。

基本的な形は次のようになります。

```rust
fn add(left: i32, right: i32) -> i32 {
	left + right
}

#[cfg(test)]
mod tests {
	use super::*;

	#[test]
	fn add_returns_sum_of_two_numbers() {
		assert_eq!(add(2, 3), 5);
	}
}
```

この最小構成のポイントは、対象コードに近い場所でテストを書くことと、`use super::*;` により同じモジュールの関数へアクセスすることです。

## まず押さえるべき内容

- テストモジュールの定義方法
- 公開関数と非公開関数のテストの考え方
- 正常系のテスト
- 異常系のテスト
- panic を期待するテスト
- 戻り値として `Result` を返すテスト

これらを学ぶときは、構文そのものよりも「どの仕様を1つのテストで守るか」を意識したほうが上達しやすくなります。

## テストモジュールの基本構成

Rustの単体テストは通常、対象コードと同じファイルの末尾に `tests` モジュールとして配置します。これには次の利点があります。

- 対象コードとテストの対応が追いやすい
- 非公開関数や内部ヘルパーも検証しやすい
- 変更時に実装とテストを一緒に更新しやすい

一方で、公開APIとしての振る舞いを確認したい場合は、単体テストだけでなく統合テストも併用したほうが設計の意図に合います。

## 正常系の書き方

正常系のテストでは、期待する入力に対して期待する出力が得られるかを確認します。ここで重要なのは、典型ケースだけで満足しないことです。

たとえば次の観点でケースを増やせます。

- 代表的な入力
- 境界値に近い入力
- 業務上よく現れる入力
- 空やゼロなどの特殊な値

正常系は「動けばよい」ではなく、「仕様の中心となる振る舞いを固定する」ためのテストです。

## 異常系の書き方

異常系では、関数が不正入力や不正状態に対してどう振る舞うかを確認します。Rustでは異常系が `Result` や `Option` によって表現されることが多いため、失敗を仕様として扱いやすいのが利点です。

たとえば、次のような点を確認します。

- エラーを返すべき条件で本当に失敗するか
- 想定した種類のエラーを返しているか
- エラー時に状態が壊れていないか
- 失敗パスで余計な副作用が起きていないか

## `panic` を期待するテスト

Rustでは `panic` を期待するテストも書けますが、これは最後の手段として考えるのが妥当です。業務ロジックでは、可能であれば `Result` を返したほうがテストしやすく、呼び出し側でも扱いやすくなります。

`panic` テストを使う場面は、主に以下です。

- 契約違反を即座に止めたいとき
- 内部不変条件が破れたときだけ失敗させたいとき
- 失敗を回復可能なエラーとして扱わない設計のとき

## `Result` を返すテスト

テスト関数自体が `Result<(), E>` を返せるため、途中で `?` を使って読みやすく書けます。複数の前処理が必要なテストでは、この書き方が有効です。

```rust
#[test]
fn parse_user_id() -> Result<(), String> {
	let value = "42".parse::<u32>().map_err(|e| e.to_string())?;
	assert_eq!(value, 42);
	Ok(())
}
```

この形は、準備処理で失敗し得るテストを簡潔に保つのに向いています。

## 正常系と異常系を分けて考える

テストでは、期待される入力に対して正しい結果を返すかだけでなく、不正入力や境界条件で適切に失敗するかも確認する必要があります。

Rustでは `Result` や `Option` が明示的に使われるため、成功と失敗の分岐をテストケースとして整理しやすい利点があります。

さらに、正常系と異常系を別のテストとして明確に分けることで、失敗したときにどの仕様が壊れたのかが見えやすくなります。1つのテストに複数の分岐を詰め込みすぎると、失敗原因の切り分けが難しくなります。

## 非公開関数をどう扱うか

単体テストでは非公開関数も検証できますが、何でも直接テストすればよいわけではありません。基本的には次の順で考えるのが堅実です。

1. まず公開関数の振る舞いとして十分に確認できないか
2. それが難しいなら、複雑な内部ロジックだけを単体テストする
3. 内部関数への依存が強すぎるなら、設計の分割を見直す

内部実装に強く結びついたテストは、リファクタリング耐性が下がります。仕様を守るテストと実装補助のテストを意識的に分けることが重要です。

## テストを書くときの観点

- 振る舞いを検証するのか、状態変化を検証するのか
- 仕様を固定したいのか、実装詳細を見ているのか
- 1つのテストで複数の責務を持ち込みすぎていないか
- 失敗したときに原因を追いやすい粒度になっているか

## よくある失敗

- テスト名から仕様が読めない
- 1つのテストで複数条件をまとめて確認している
- 実装詳細に依存しすぎて、リファクタリングで壊れやすい
- 正常系ばかりで失敗パスを確認していない
- 準備処理が長すぎて、何を検証しているか分からない

## この章の要点

- 単体テストは小さな責務の仕様を固定するために書く
- `#[cfg(test)]` と `#[test]` が基本の出発点になる
- 正常系、異常系、境界値を分けて考えると漏れが減る
- `panic` より `Result` のほうが扱いやすい場面は多い
- 実装詳細ではなく守りたい仕様を中心にテストを設計する

単体テストは、単に数を増やせばよいわけではありません。読みやすく、失敗理由が明確で、仕様を保護できる粒度で作ることが重要です。
