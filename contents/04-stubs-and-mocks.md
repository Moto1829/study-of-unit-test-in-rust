# スタブやモックを使ったテスト

外部サービス、時刻、乱数、ファイルI/O、データベースなどに直接依存したコードは、そのままだと単体テストしづらくなります。Rustではtraitを使って依存を抽象化し、テストではスタブやモックに差し替える設計が有効です。

ここで重要なのは、スタブやモックは単なるテストテクニックではなく、依存関係を整理した設計の副産物でもあるという点です。つまり、テストしづらいコードは、しばしば責務や境界の切り方にも問題を抱えています。

## スタブとモックの役割

スタブは、テスト用に決め打ちの値や結果を返すための代替実装です。分岐の確認や失敗ケースの再現に向いています。

モックは、戻り値を返すだけでなく、呼び出し回数や引数などの相互作用も検証したいときに使います。

整理すると、次のように使い分けられます。

- スタブ: 入力に対して決めた結果を返し、分岐を確認したい
- モック: 依存先とのやり取り自体が仕様の一部になっている

たとえば、通知送信が成功したかどうかだけを見たいならスタブで十分です。一方、通知を1回だけ送ること自体が要件なら、モックによる呼び出し検証が有効です。

## Rustで依存を切り分ける基本戦略

- trait による依存の抽象化
- 実装をコンストラクタ引数などから注入する
- 本番実装とテスト実装を明確に分離する
- テストで扱いやすい最小限のインターフェースに絞る

ここで特に重要なのは、traitを大きくしすぎないことです。大きすぎるtraitはテストダブルの実装コストを上げ、結果としてテストしにくくなります。

## 依存抽象化の簡単な例

```rust
trait UserRepository {
	fn find_name(&self, user_id: u64) -> Option<String>;
}

struct GreetingService<R: UserRepository> {
	repository: R,
}

impl<R: UserRepository> GreetingService<R> {
	fn greeting_for(&self, user_id: u64) -> String {
		match self.repository.find_name(user_id) {
			Some(name) => format!("Hello, {}", name),
			None => "Hello, guest".to_string(),
		}
	}
}
```

この形にしておくと、`UserRepository` を実装したスタブを用意するだけで、データベースに依存せず振る舞いを確認できます。

## なぜtraitが重要か

Rustでは、依存対象をtraitで表現しておくことで、実装詳細に縛られずに振る舞いを差し替えられます。これはテストのためだけではなく、設計の境界を明確にする効果もあります。

さらに、traitを使うことで「このロジックが外部に何を期待しているのか」が型として表面化します。これはテスト容易性だけでなく、責務分離の明確化にも効きます。

## テストで確認したいこと

- 固定値を返したときに分岐が期待通り動くか
- エラーを返したときに呼び出し元が適切に処理するか
- 依存先の呼び出し回数が過不足ないか
- 意図した引数で依存先を呼んでいるか

## スタブで十分な場面

次のような場面では、まずスタブで考えるのが妥当です。

- 条件分岐の結果だけを確認したい
- エラー時の処理を再現したい
- 外部依存を完全に排除したい
- 相互作用の詳細までは仕様に含まれない

スタブはシンプルなので、テストの意図が読みやすくなりやすい利点があります。

## モックが必要な場面

一方で、次のような場合はモックの価値があります。

- 送信や保存の回数に意味がある
- 呼び出し順序が仕様に影響する
- どの引数で呼んだかを保証したい
- 副作用の実行有無そのものを確認したい

ただし、呼び出し回数や順序の検証は実装詳細に近づきやすいため、本当に仕様として守るべきなのかを毎回確認したほうがよいです。

## よくある設計上の問題

- 依存先traitが大きすぎる
- 具体型に直接依存していて差し替えられない
- 1つのユースケースが複数外部依存に強く結びついている
- 時刻や乱数が隠れておらず、テストが不安定になる

こうした問題がある場合、スタブやモックを増やす前に責務分割を見直したほうが結果的にシンプルになります。

## この章の要点

- スタブは分岐確認、モックは相互作用検証に向く
- traitによる抽象化はテスト技法であると同時に設計技法でもある
- 差し替え可能な依存を作ると単体テストが安定する
- モックの使いすぎは実装依存を強める
- テストしづらいなら、まず設計の境界を疑う

スタブやモックは便利ですが、使いすぎると実装詳細に依存しすぎたテストになりがちです。何を守りたい仕様なのかを基準に使い分ける必要があります。
